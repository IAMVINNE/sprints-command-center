Documentação Técnica — Sprint Command Center

Versão: 1.0.0  |  Data: Abril 2026  |  Autor: Vinícius Santos, PO Autoatendimento CX
Área: Autoatendimento Customer Experience  |  Classificação: Documento interno — Uso restrito à área

1. Visão Geral

O Sprint Command Center é uma interface web estática desenvolvida para centralizar a visualização e o acompanhamento de cards de sprint da squad de Autoatendimento Customer Experience. A plataforma foi concebida para oferecer à coordenação (e demais stakeholders) uma visão panorâmica e analítica do backlog, status de entregas, distribuição de esforço, uso de IA e indicadores de bloqueio — tudo a partir de uma planilha Excel mantida pelo Product Owner.

A solução foi projetada como uma ferramenta de produtividade pessoal do PO, sem integração com sistemas internos do banco e sem qualquer acesso, armazenamento ou transmissão de dados sensíveis ou confidenciais.

2. Arquitetura Técnica

2.1 Stack Tecnológica

A plataforma é composta por um único arquivo HTML autocontido que utiliza exclusivamente tecnologias front-end padrão da web:

  • HTML5 — Estrutura semântica do documento
  • CSS3 — Estilização completa (variáveis CSS, grid, flexbox, animações)
  • JavaScript ES6+ — Lógica de processamento, filtros, renderização
  • Chart.js v4.4.0 — Biblioteca open-source para gráficos (licença MIT)
  • chartjs-plugin-datalabels v2.2.0 — Plugin para rótulos em gráficos (licença MIT)
  • SheetJS (xlsx) v0.18.5 — Biblioteca open-source para leitura de arquivos Excel (licença Apache 2.0)
  • Google Fonts — Tipografia (DM Sans, Space Grotesk, JetBrains Mono)

2.2 Arquitetura de Execução

A aplicação opera inteiramente no navegador do usuário (client-side). Não existe backend, servidor, banco de dados, API ou qualquer componente de infraestrutura associado. O arquivo HTML é aberto localmente e todas as operações — leitura de planilha, processamento de dados, renderização de gráficos, aplicação de filtros — são executadas pelo motor JavaScript do navegador.

Fluxo de dados: Arquivo Excel (.xlsx) na máquina local → Upload via drag-and-drop no navegador → Processamento via SheetJS em memória RAM do navegador → Renderização visual via Chart.js e DOM → Dados permanecem exclusivamente na memória volátil do navegador durante a sessão → Ao fechar a aba/navegador, todos os dados são descartados automaticamente.

2.3 Dependências Externas (CDN)

As bibliotecas são carregadas via CDN (Content Delivery Network) pública no momento da abertura do arquivo. Estas são bibliotecas open-source amplamente utilizadas pela comunidade de desenvolvimento web:

  • cdnjs.cloudflare.com — CDN da Cloudflare (SheetJS)
  • cdn.jsdelivr.net — CDN do jsDelivr (Chart.js)
  • fonts.googleapis.com — Google Fonts (tipografia)

Nenhuma dessas requisições transmite dados do usuário. São downloads unidirecionais de bibliotecas JavaScript e fontes.

3. Segurança e Privacidade

3.1 Isolamento Total de Dados

DECLARAÇÃO DE SEGURANÇA: O Sprint Command Center NÃO acessa, coleta, armazena, transmite ou processa dados de nenhum sistema, banco de dados, API ou infraestrutura interna ou externa. Todos os dados exibidos na interface são provenientes exclusivamente de um arquivo Excel fornecido manualmente pelo usuário, processados em memória volátil do navegador, e descartados integralmente ao encerrar a sessão.

3.2 O que a plataforma NÃO faz

  • NÃO se conecta a nenhum banco de dados (interno ou externo)
  • NÃO realiza chamadas de API para servidores internos do banco
  • NÃO acessa cookies, localStorage ou sessionStorage para persistir dados
  • NÃO transmite dados via rede (nenhum POST, PUT ou envio de formulário)
  • NÃO executa coleta de dados do usuário (analytics, tracking, telemetria)
  • NÃO acessa o sistema de arquivos do computador além do arquivo selecionado pelo usuário
  • NÃO se comunica com servidores da Anthropic (Claude) após a geração do código
  • NÃO armazena dados entre sessões — cada abertura do arquivo é uma sessão limpa

3.3 O que a plataforma FAZ

  • Lê um arquivo Excel selecionado manualmente pelo usuário via interface do navegador
  • Processa os dados desse arquivo em memória JavaScript (RAM do navegador)
  • Renderiza gráficos e tabelas visualmente no DOM do navegador
  • Carrega bibliotecas open-source via CDN (requisição GET somente de download)
  • Descarta todos os dados automaticamente ao fechar a aba ou o navegador

3.4 Compatibilidade com Políticas Corporativas

Por se tratar de um arquivo HTML estático executado localmente, a plataforma se comporta de forma análoga a um documento Excel com macros VBA ou um arquivo PDF interativo — ferramentas de produtividade que operam inteiramente no ambiente local do usuário sem interação com infraestrutura corporativa.

A plataforma não requer instalação, não modifica configurações do sistema, não solicita permissões administrativas e não interage com nenhum sistema bancário.

4. Governança de Dados

4.1 Armazenamento

NENHUM DADO É ARMAZENADO EM NENHUM LOCAL. Os dados existem exclusivamente na memória RAM do navegador durante a sessão ativa. Não há persistência em disco, nuvem, banco de dados, cache ou qualquer outro mecanismo de armazenamento. Ao fechar o navegador ou a aba, os dados são irrecuperavelmente descartados pelo garbage collector do JavaScript.

4.2 Fluxo de Dados Detalhado

  • Origem: Planilha Excel (.xlsx/.xls/.csv) criada e mantida pelo PO na máquina local
  • Ingestão: Upload via File API do navegador (seleção manual ou drag-and-drop)
  • Processamento: Parsing via SheetJS → conversão para array de objetos JavaScript em memória
  • Visualização: Renderização de KPIs, gráficos (Chart.js) e tabelas (DOM manipulation) no navegador
  • Filtragem: Operações de Array.filter() sobre os dados em memória
  • Descarte: Automático ao fechar a sessão — sem salvamento, sem persistência

4.3 Dados Sensíveis

A planilha de input contém exclusivamente dados operacionais de gestão de sprint (nomes de cards, status, story points, nomes de desenvolvedores). Não há manipulação de dados de clientes, dados financeiros, dados pessoais sensíveis (LGPD), credenciais ou informações classificadas. O conteúdo da planilha é de controle exclusivo do PO e reflete informações já disponíveis em ferramentas de gestão de projetos da squad.

5. Sobre o Desenvolvimento

5.1 Processo de Criação

O código-fonte foi gerado com auxílio de inteligência artificial (Claude, Anthropic) sob orientação direta e especificação funcional do Product Owner Vinícius Santos. O processo de desenvolvimento seguiu o modelo de "AI-assisted development", onde:

  • O humano (PO) definiu todos os requisitos funcionais, regras de negócio, estrutura de dados, identidade visual e critérios de aceitação
  • A IA (Claude) gerou o código-fonte baseado nessas especificações
  • O humano revisou, testou, validou e solicitou ajustes iterativamente
  • O produto final é de propriedade intelectual do autor humano, que detém total responsabilidade sobre seu uso e manutenção

5.2 Relação com a Anthropic (Claude) Pós-Geração

Após a geração do código, não existe nenhuma conexão, comunicação ou dependência entre esta plataforma e os servidores da Anthropic/Claude. O código gerado é um artefato estático — equivalente a um documento ou planilha. A Anthropic não tem acesso ao código em uso, aos dados processados, nem às interações do usuário com a plataforma. Não há telemetria, callbacks, webhooks ou qualquer canal de comunicação com a Anthropic.

5.3 Legitimidade do Desenvolvimento Assistido por IA

O uso de ferramentas de IA para geração de código é uma prática amplamente adotada na indústria de tecnologia (GitHub Copilot, Amazon CodeWhisperer, Cursor, etc.). O código gerado por IA é equivalente a código gerado por qualquer outra ferramenta de desenvolvimento — a responsabilidade, propriedade intelectual e controle são do desenvolvedor humano que especificou, revisou e implantou a solução.

Este desenvolvimento não viola nenhuma norma de propriedade intelectual, uma vez que o produto final reflete especificações originais do autor e não reproduz código proprietário de terceiros.

6. Registro de Eventos (Audit Log)

A plataforma mantém um log em memória (volátil) de todas as ações realizadas durante a sessão: carregamento de planilha, aplicação de filtros, troca de visualizações e exportações. Este log é visível na aba "Eventos" e é descartado ao fechar o navegador, seguindo a mesma política de não-persistência de dados da plataforma.

7. Limitações Conhecidas

  • Dados não persistem entre sessões — a planilha deve ser recarregada a cada uso
  • Requer conexão com internet na primeira abertura (carregamento de bibliotecas CDN)
  • Capacidade limitada ao volume de dados que o navegador consegue processar em memória
  • Não substitui ferramentas formais de gestão de projetos (Jira, Azure DevOps, etc.)
